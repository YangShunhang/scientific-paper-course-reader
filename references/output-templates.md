# 输出模板

## 启动目录模板

正式精读前，先输出：

```markdown
我会把这篇论文按课程方式讲解。先给出阅读路线。

**论文类型**：...
**难度判断**：...
**推荐节数**：...
**是否建议分两次讲**：...

**课程目录**
1. ...
2. ...
3. ...
```

目录要简洁，不要在这里展开大量解释。

## 标准版每节模板

```markdown
## 第 X 节：标题

**本节要解决的问题**  
...

**关键概念**  
- **英文名（缩写，中文名）**：一句通俗解释；说明它在本文中的作用。

**核心解释**  
...

**公式、图或实验讲解**  
...

**本节小结**  
...
```

如果没有关键概念、公式、图或实验，对应部分可以省略。对于核心方法、核心模块、关键训练机制、关键实验等章节，应使用下面的“核心模块节模板”思路展开，不要只套用标准版轻量结构。

## 核心模块节模板

当本节讲论文核心模块、核心算法、关键训练策略、关键增强方法、架构分支、核心系统组件或关键实验设计时，使用更详细的讲解结构。优先参考以下角度，并根据论文内容选择真正相关的部分展开；不要机械逐条套用，也不要为了套模板引入论文中不存在的问题。

```markdown
## 第 X 节：模块名：它为什么是论文核心

**本节要解决的问题**  
说明该模块或设计解决前文哪个具体困难，为什么它不是可有可无的细节。

**先讲直觉**  
用初学者能理解的方式解释：没有这个设计时，方法链条、实验论证或应用场景会缺什么。

**机制拆解**  
按输入 → 中间处理 → 输出讲清每一步。不要只复述论文名词。

**和前后内容的关系**  
说明它接收谁的结果、影响谁、是否参与训练/推理、是否改变目标函数、表示空间、系统流程或实验结论。

**关键公式、图或实验证据**  
如果有公式，解释每个符号、每一项作用、参数变大/变小时的影响。  
如果有图，解释数据流、控制流或信息流。  
如果有实验或消融，说明它如何支持该设计。

**为什么这个设计可能有效**  
从论文实际内容出发，结合建模、优化、表示学习、统计稳定性、工程约束、理论条件或实验现象解释。

**如果去掉它会怎样**  
结合消融、直觉或作者证据说明可能后果；如果论文没有证据，应说明证据不足。

**本节小结**  
用几句话说明这个模块对论文贡献的真实作用。
```

## 分段输出模板

如果论文过长或复杂，先讲前半部分。结尾使用：

```markdown
本次先讲到这里。到目前为止，你已经理解了：

1. ...
2. ...
3. ...

下一部分建议继续讲：

- 核心实验设计；
- 主要结果和消融；
- 创新与局限；
- 复现和改进建议。
```

不要承诺后台继续完成。等待用户继续指令。

## 重点公式总表

全文讲完后输出：

```markdown
| 公式/编号 | 所在部分 | 作用 | 核心直觉 | 是否影响论文主线 |
|---|---|---|---|---|
| ... | ... | ... | ... | 是/一般 |
```

只收录重点公式。

## 重点图片总表

```markdown
| 图号 | 所在部分 | 内容 | 读图重点 | 与方法/实验的关系 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |
```

只收录关键图。

## 实验结果总表

```markdown
| 实验/指标 | 比较对象 | 主要结果 | 相对最佳基线 | 解释 | 可信度注意点 |
|---|---|---:|---:|---|---|
| ... | ... | ... | ... | ... | ... |
```

如果论文没有足够数值，不要编造。说明“论文未明确报告”。

## 最终知识地图

用一句话或一条链路表达：

```markdown
问题 → 建模/假设 → 方法 → 训练/推理 → 实验 → 结论与局限
```

示例：

```markdown
这篇论文的知识地图是：它从 ... 问题出发，先把 ... 建模为 ...，再用 ... 方法解决 ...，最后通过 ... 实验证明 ...，但仍受 ... 限制。
```

## 深度控制

默认不要为了简短而牺牲理解链路。控制目标不是“短”，而是“该详细的地方详细，不重要的地方不堆砌”。

区分三种讲解深度：

1. **核心主线内容**：详细讲。包括它为什么出现、解决什么问题、内部机制、和前后内容的关系、作者证据以及初学者容易误解的点。
2. **支撑性内容**：中等讲。说明作用、必要背景和与主线的连接。
3. **边缘内容**：略讲。只说明它为什么不影响主线，或直接省略。

不要把所有章节写成同样长度。重要章节可以明显更长。

判断篇幅时参考：标题、摘要、贡献列表、方法结构、架构图、公式位置、实验设计、消融实验和作者反复强调的概念。最终要根据具体论文再做判断。

如果某一节是论文贡献核心，即使用户没有单独要求，也要主动展开；用户要求“详细讲第 X 节”时，再进一步补充推导、例子、类比和可复现细节。

## 不合格输出模式

避免以下输出：

1. 只说“这个模块提升性能、增强鲁棒性、捕捉关系或改善表示”，但不解释它具体改变了什么信息、流程、约束或归纳偏置。
2. 只列组件名称，不解释输入、输出、中间处理和信息流。
3. 只翻译作者术语，不说明为什么需要这个设计。
4. 只给公式，不解释每一项的作用、符号含义和参数变化影响。
5. 对核心贡献和普通背景使用同样篇幅。
6. 在核心模块还没拆开前就直接给“核心直觉”或“小结”。
7. 把已有基础模块、常规训练技巧或标准实验设置误写成作者原创贡献。
8. 为了套模板引入论文没有讨论的问题、假设或机制。
